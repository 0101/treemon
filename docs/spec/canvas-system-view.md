# Canvas System Views — Separating Beads from Agent Docs

> Status: **Shipped / implemented** (`tm-canvas48-0su`, verified end-to-end by `tm-canvas48-klm`). Captures the architecture that separated the beads dashboard from the agent-doc machinery, which came out of debugging the beadspace 30s state-reset (see `docs/spec/canvas-pane.md`, `docs/spec/beadspace-canvas.md`). The "Approach" below is now the as-built design; the **Decisions** and **Resolved Questions** sections carry the concrete implementation and the task IDs that delivered each piece.

## Problem

The beads dashboard (`beads.html`) is **auto-generated by the server** and renders in the canvas pane. Originally it flowed through the pane as a plain `CanvasDoc` with **zero special-casing in `CanvasPane.fs`**. As a result it inherited the entire generic "canvas document" machinery — including the parts that were designed for a fundamentally different thing.

Two distinct concepts were collapsed into one (`CanvasDoc`):

| | **Agent doc** (e.g. a status page, a form) | **Beads dashboard** (a generated view) |
|---|---|---|
| Created by | a coding session writing `.html` | the server, from `BeadspaceTemplate` |
| Owner | the author session (`OwnerSessionId`) | none |
| Where content lives | in the file | in `.beads/beads.db`, fetched via `/beads-data` |
| How it updates | file overwrite → hash change → **morph** | its own 30s data poll + in-place refresh |
| Interactivity | posts payloads back to its owner session | none (only link-nav + self `refresh-beads`) |

### What the beads dashboard actually uses vs. inherits

Layer-by-layer (full diagram was shared in the canvas pane during the session):

- **Genuinely shared (keep):** scan + hash (`CanvasScanner`), serve + inject (`CanvasDocServer` @ :5002), the **pane shell** (tabs, iframe, docking, archive, overview), disk-as-source-of-truth.
- **Beads-specific (already special, fine):** auto-provisioning (`BeadspaceProvisioner`) and the private `/beads-data` JSON endpoint.
- **Inherited but a misfit (the problem):**
  - **Liveness dot** — always shows "dead" because there is no owner session.
  - **`▶ Start session` button** — appears for a generated view, where starting a session is meaningless.
  - **Message bridge** (postMessage → session `/inject`, queue) — unused; there is no author session to route to.
  - **Awareness** (unviewed badge / card notification / idle auto-display keyed on `contentHash`) — largely inert, because the file hash is stable while the *data* changes. Beads "newness" already lives on the worktree card as `BeadsSummary`.
  - **Morph (Level 3)** — redundant with the dashboard's own refresh, and actively harmful when it does fire (a deploy/template change morphs the live, JS-rendered dashboard back down to the empty template shell).

The session-document layer exists for "an interactive document authored and owned by a live session." A beads dashboard is **none of those** — it is a system-generated, data-driven view that only wants the UI real-estate.

## Goals

- Model the two concepts distinctly so misfit behaviors become **unrepresentable** rather than emergent from `OwnerSessionId = None`.
- Give the beads dashboard a **distinct UI affordance** (it should not look like, or sit among, agent doc tabs).
- Preserve the full agent-doc experience (ownership, liveness, Start-session, message bridge, morph, awareness) for genuinely agent-authored docs.
- No regressions to the beadspace fixes already shipped (template sync + unchanged-payload skip-guard).

## Non-Goals

- Removing morph / Level 2–4 — they are valuable for agent docs (interactive, file-driven, overwrite-in-place). They simply should not apply to system views.
- Re-theming or restructuring the beadspace dashboard internals.

## Approach (as built)

### 1. Model the kind (domain-driven, repo-idiomatic)

A `CanvasDocKind` was added to the canvas doc domain type (`src/Shared/Types.fs`):

```fsharp
type CanvasDocKind =
    | AgentDoc      // authored & owned by a session; interactive; file-driven
    | SystemView    // server-generated; data-driven; no owner (e.g. the beads dashboard)
```

- `CanvasScanner.scan` tags every scanned doc via `CanvasDocKind.classify filename` — the single, shared classifier (`beads.html` → `SystemView`, everything else → `AgentDoc`). It is the sole non-test construction site, and the one place to register future generated views; the typed `Kind` field is then serialized to the client so no consumer sniffs filenames.
- The session-document machinery is gated on `AgentDoc`:
  - Liveness dot, `▶ Start session`, message-bridge routing, and **morph signaling** apply to `AgentDoc` only.
  - `SystemView` docs are excluded from content-hash awareness entirely (the card's `BeadsSummary` already covers "something changed").

This makes the misfit states impossible to construct, instead of relying on `None` defaults that happen to render as "dead."

### 2. Distinct UI for system views

- Beads presents as a **dashboard affordance**, not a doc tab — a distinct entry pinned to the far left of the tab strip (class `.canvas-system-tab`, a "BD" glyph + issue-count badge), visually separated from the agent-doc tabs.
- A `SystemView` keeps: the iframe, docking/position, and the pane real-estate. It drops: liveness dot, Start-session, the doc-style tab treatment, and the archive button.
- **Resolved:** the beads view lives *inside* the same pane tab strip (styled differently, pinned far-left), rather than being a separate header toggle — see Resolved Questions.

### 3. Morph scope

- Morph (and the idiomorph injection) applies to `AgentDoc` only. `SystemView` docs are served *without* the idiomorph runtime or the morph controller at all — `CanvasDocServer.buildInjection SystemView` emits `baseStyle + linkInterceptor` only — and the client never emits a `MorphActiveDoc` signal for them.
- This dissolves the earlier "morph vs beadspace" conflict entirely: beads opts out by virtue of its kind.

## Decisions (carried in from the session)

- **Template sync (shipped):** `beads.html` is regenerated whenever its on-disk content differs from `BeadspaceTemplate.html`, so template fixes propagate on deploy. It is a generated view, not user-customized.
- **Unchanged-payload skip (shipped):** the dashboard's 30s poll skips re-render when `/beads-data` is byte-identical to the prior fetch.
- **Beads canvas = All Issues table only (shipped, `tm-canvas48-dsh`):** the beads `SystemView` renders only the "All Issues" table. The second "Dashboard" view (stat cards, status donut, priority/label/type charts, triage + active-issues panels) and the entire top navigation bar were removed. The single issues view is always visible (`.view { display: block }` — no nav/`.active` toggle), and the detail panel, status/search/filter controls, 30s poll, unchanged-payload skip, and postMessage refresh are all retained.
- **`.fs`/`.html` byte-identical reconciliation (`tm-canvas48-dsh`):** `BeadspaceTemplate.fs` (the `[<Literal>]` served at runtime) and `BeadspaceTemplate.html` (loaded from disk by the E2E tests) had silently diverged — the `tm-canvas48-h8s` `addBadge` detail-panel refactor had landed only in `.fs`. Reconciliation rule going forward: make edits to `.html`, converge the detail panel to the `addBadge` helper, then regenerate `.fs` as `module Server.BeadspaceTemplate` + `let html = """` + the exact `.html` bytes + `"""` so the literal's content is byte-identical to the file (CRLF throughout, no BOM, no `"""` inside the content). No automated test enforces this equality yet; `BeadspaceProvisionerTests` only round-trips the literal.
- **Separate, don't gut:** agent-authored interactive docs are a real use case, so the session-document layer stays — beads is carved out of it.
- **Server injection gated by kind (shipped, `tm-canvas48-dhl`):** `CanvasDocServer.handleCanvasRequest` classifies the served filename via `CanvasDocKind.classify` and builds the `<head>` injection via the new `CanvasDocServer.buildInjection`. Both kinds keep `baseStyle` + `linkInterceptor`. `AgentDoc` keeps the full injection (`bridgeScript` heartbeat + `IdiomorphScript.idiomorphJs` + `IdiomorphScript.morphController`); `SystemView` (e.g. `beads.html`) omits all three. Rationale: a system view must never morph (a morph stomps the live, JS-rendered dashboard back to the empty template shell), and — the one judgment call the task left open — `bridgeScript` is also dropped because a system view has no owner session to route bridge messages to. Safe because the dashboard drives itself via its own 30s `/beads-data` poll and a direct `refresh-beads` postMessage listener, neither of which depends on the bridge heartbeat (which only dispatches `canvasBridgeMessage` events the dashboard never consumes). Covered by `CanvasDocServerTests` (`buildInjection`/`classify` mapping); the end-to-end HTTP proof against :5002 is owned by the `tm-canvas48-klm` verify task.
- **Client session-document machinery gated by kind (shipped, `tm-canvas48-fj9`):** the client mirrors the server gating so misfit affordances never render for a `SystemView`. `CanvasPane.fs` routes both liveness-dot call sites (tab strip + overview) through `livenessDotFor`, which returns `Html.none` for `SystemView`, and gates the `▶ Start session` button on `doc.Kind = AgentDoc`. `App.fs` adds a shared `canvasDocKind repos scopedKey filename` lookup and uses it to (a) drop `SystemView` docs from idle auto-display candidates so a beads data-poll never focus-steals, and (b) suppress the `MorphActiveDoc` signal for `SystemView` active docs. Judgment call: the task scoped morph-gating to the data-poll `morphCmd`, but the same `Cmd.ofMsg MorphActiveDoc` is also emitted on tab re-select (`SelectCanvasDoc`, `wasAlreadyVisited`), so **both** emit sites were gated to fully realize "morph applies to AgentDoc only" on the client (the re-select morph was already inert for `SystemView` since dhl injects no morph controller, so this only removes a redundant no-op postMessage). Covered by three E2E render assertions in `CanvasPaneTests` against the multirepo fixture (beads `SystemView` + dashboard/status `AgentDoc`): SystemView shows no liveness dot (tab + overview) and no Start-session, AgentDocs still show both.
- **Awareness excludes SystemView at the data source (shipped, `tm-canvas48-45p`):** content-hash awareness must ignore `SystemView` docs — the beads file hash is stable while its data changes (so it would never signal real newness) and would morph-signal spuriously on a template/deploy edit; beads "newness" already lives on the worktree card as `BeadsSummary`. Implemented as a single private `awarenessDocs = List.filter (fun d -> d.Kind = AgentDoc)` chokepoint in `CanvasAwareness.fs`, routed through `seedLastViewedHashes`, `unviewedDocsByScopedKey`, `canvasHashesByScopedKey`, and `findMostRecentChangedDoc`. `detectCanvasEvents` needs no change of its own: it derives events purely from the hash maps, and since `canvasHashesByScopedKey` never emits a `SystemView` hash, a beads data-change can't produce a `NewDoc`/`UpdatedDoc` event. This also makes `findMostRecentChangedDoc` self-defending — `App.fs` already filters auto-display candidates to `AgentDoc` (fj9), but the function now ignores `SystemView` regardless of caller. Covered by `SystemViewAwarenessTests` in `CanvasAwarenessTests.fs` (11 tests across all four functions, proving a `SystemView` doc never contributes to unviewed counts, canvas events, seeded hashes, or the idle auto-display target). Note: `CanvasAwarenessTests` has 9 pre-existing failures (4 `Auto-display*` + 5 `MarkDocViewed*`) from the `App` `.cctor` building a Fable.Remoting client proxy under the .NET test host — unrelated to awareness, documented in `tm-canvas48-cc4`; the new tests avoid all `App` module bindings and so are immune.
- **Distinct SystemView affordance + archive gating (shipped, `tm-canvas48-86v`):** the `SystemView` (beads) now renders as a distinct entry in the tab strip via a new private `CanvasPane.systemViewTab` — class `.canvas-system-tab` (NOT `.canvas-tab`), a `"BD"` glyph plus a count badge (`.canvas-system-tab-count`), no liveness dot. The label is the worktree's total beads issues `Open + InProgress + Blocked + Closed` (helper `beadsTotal`, sourced from the in-scope `wt.Beads`), replacing the dashboard's former top-bar "N issues" meta. It is pinned far-left by a stable `List.sortBy` keying `SystemView → 0`/`AgentDoc → 1` over `wt.CanvasDocs`. **Gate change:** the tab strip now renders when `wt.CanvasDocs.Length > 1 || hasSystemView` (was `> 1`), so a lone `beads.html` still shows its count badge; a lone `AgentDoc` still shows no tabs. The **archive button** is gated in `headerBar` by widening its parameter from `activeFilename: string option` to `activeDoc: CanvasDoc option` and rendering the button only for `Some d when d.Kind = AgentDoc` (a `SystemView` is server-regenerated, not user-owned). CSS lives in `index.html` (`.canvas-system-tab*`, no inline styles). Judgment call: the fj9 test `SystemView tab shows no liveness dot…` located the beads entry via `.canvas-tab` HasText "beads", which my change makes a no-op (beads is no longer a `.canvas-tab` and its label is the count, not "beads"); its `beadsTab` locator was repointed to `.canvas-system-tab` to keep the assertion honest. Covered by three new `CanvasPaneTests` E2E assertions against the multirepo fixture (beads total = 2): distinct leftmost non-`.canvas-tab` entry with no liveness dot, count badge shows "2", archive hidden while the SystemView is active but present after switching to an `AgentDoc`. Verified: full build 0/0; Fable compile clean; 37/38 Canvas E2E pass (the lone failure, `Liveness dot renders in overview entries`, is the pre-existing `.canvas-overview-branch` selector bug from cc4, untouched here).
- **E2E fixture-ordering regression fixed (`tm-canvas48-76g`):** the cc4 fixture addition (`feature-multidoc` in `src/Tests/fixtures/worktrees.json`, given the most-recent `LastCommitTime` and no active session) made it the first card under the dashboard's default `ByActivity` sort, flipping the first `.terminal-btn`'s title to "Open terminal (Enter)". That broke the `Category=E2E` test `DashboardTests."Header button has correct text and title"(terminal-btn)`, which asserted the *first* button (`.wt-card .terminal-btn .First`). Product title logic (`App.fs:1290`, unchanged) is correct; the fix re-targets the assertion at a card with a known session state — `.wt-card.has-session .terminal-btn` (the `has-session` class is set by `cardClassName` when `HasActiveSession=true`) — so future fixture reordering can't flip it. Lesson: tests keying off `.First` of an activity-sorted list are fragile to fixture additions; assert against a stable state class instead. CI-scope note: `Category=E2E` sits outside the local Fast/Canvas gates but *inside* CI (`dotnet test --filter "Category!=Local"`), so this would have turned branch CI red while passing the local gates.
- **Local-only App `.cctor` test failures fixed (`tm-canvas48-re8`):** the ~76 `Category=Fast` tests that failed ONLY on the local Windows .NET test host with `TypeInitializationException` from `App`'s module initializer (they already pass on CI/Linux under `--filter Category!=Local`) now pass. The root cause was not one but **three sequential init-time landmines** in `App.fs`, each masking the next — because module init runs every top-level binding in source order, touching ANY `App` member triggers the whole `.cctor`: (1) `worktreeApi = Remoting.createApi() |> Remoting.buildProxy<IWorktreeApi>` (~line 99) — `Fable.Remoting.Client.buildProxy` uses reflection that throws `ArgumentException` under the Windows .NET runtime (fine under Fable/JS and on Linux .NET); (2) six module-level Feliz SVG icon **values** (`editorIcon`, `resumeIcon`, `binIcon`, `conflictIcon`, `viewEyeRolledBack`, `viewEyeClosed`), each throwing `InvalidCastException` (Tuple2→`ISvgAttribute`) under .NET; (3) the top-level `Program.mkProgram init update view |> … |> Program.run` bootstrap, which runs the whole app (`Date.now()`, React DOM mount) at module load. **Fixes (all in `App.fs`; the Fable/browser build is byte-identical):** (1) `worktreeApi : Lazy<IWorktreeApi> = lazy (…)` with its ~29 use sites reading `.Value` — the `Lazy<IWorktreeApi>` pattern is already proven Fable-compatible in `ArchiveViews.fs`/`CreateWorktreeModal.fs`; (2) the six icons converted to `()`-thunks (`let editorIcon () = …`, each single use site → `editorIcon ()`), DOM-identical under React; (3) the bootstrap + `open Elmish.React` wrapped in `#if FABLE_COMPILER … #endif` (FABLE_COMPILER is always defined by Fable and never by .NET, so the Vite entry point is unchanged). **Test-side coordination (required adjustment, not coverage loss):** deferring the proxy to first `.Value` access changes the proxy-build failure from a `TypeInitializationException` (thrown by the eager `.cctor`) into a raw `ArgumentException` (a lazy thunk is not a type initializer). Three pre-existing helpers that already tolerate this exact defect by catching `TypeInitializationException` and re-deriving the model — `tryUpdateModel` in `CreateWorktreeTests.fs`, `ConfirmModalTests.fs`, and `CanvasAwarenessTests.fs` — were extended to `| :? TypeInitializationException | :? ArgumentException ->`. This preserves their existing `| _ -> reraise()` specificity (only the explicitly-handled msg types are tolerated) and keeps the tests fully meaningful on CI/Linux where no exception is thrown; without it the ~17 tests that **passed** in baseline via the catch would have regressed. **Result:** `Category=Fast` 722→**798 passed**, 89→**13 failed**, `TypeInitializationException` 228→**0**; build 0/0. The remaining 13 are the pre-existing `Archive … [5 s]` Playwright/browser E2E timeouts (`ArchiveTests.fs`, `Microsoft.Playwright.NUnit`) that need a running server/browser — environmental, untouched by this task. Supersedes the cc4/45p note that treated these as merely "documented, unrelated".
- **Stale overview liveness-dot selector fixed (`tm-canvas48-9hj`):** the lone Canvas E2E failure flagged in the 86v note — `CanvasPhase4Tests."Liveness dot renders in overview entries"` — was a stale **test** selector, not a product defect. The fj9/dst.7 overview restructure renders each doc's liveness dot via `livenessDotFor` *inside* the per-doc `.canvas-overview-doc` span (`CanvasPane.fs:117-122`), while the `.canvas-overview-branch` span (`:106`) holds only the branch-name text. Against the fixture the corrected selector matches **6** dots (feature-active ×1 + feature-multidoc ×3 + multirepo dashboard/status ×2; beads SystemView contributes none). Verified: build 0/0; `Category=Canvas` **47/47 pass** (was 46/47, this test the sole failure). Lesson: when a render is restructured, grep the E2E suite for every selector keyed off the old container class — the dst.7 move updated the product but left this descendant selector pointing at the wrong ancestor.
- **Stale Archive E2E mock JSON fixed (`tm-canvas48-l6o`):** the 13 `Archive*` `Category=E2E,Fast` failures the `re8` note logged as "pre-existing Playwright/browser timeouts… environmental" were **not** environmental — they were a branch-introduced regression. `ArchiveTests.fs`'s mock JSON (`makeWorktreeJson`/`makeDashboardJson`, served via Playwright `RouteAsync`) had gone stale against the new required `Shared.Types` fields this branch added (`BeadsSummary.Blocked`, `WorktreeStatus.CanvasDocs`, `DashboardResponse.CanvasPosition`), so Fable JSON decoding threw, **no** `.wt-card`/`.archive-card` ever rendered, and the selectors timed out at 5 s. `origin/main` stays green because those fields don't exist there (the decoder silently ignored the stale `"CanvasPosition":"right"` as an unknown key); on the branch `CanvasPosition` is now a real DU field, so unfixed this would also have turned **branch CI** red (`Category=E2E` ⊂ `--filter Category!=Local`). Fix is test-only — three edits to satisfy the decoders: add `"Blocked":0` to the Beads object, add `"CanvasDocs":[]` to the worktree object, change `"CanvasPosition":"right"`→`"Right"`. **Convention:** fixture/mock JSON must spell out every field a decoder requires — list fields explicit (`CanvasDocs:[]`), nullary DU cases as their exact case name (`"Right"`, like the existing `"Idle"`/`"NoPr"`); only genuine `option` fields absent on main (`Provider`, `DeployBranch`, `SystemMetrics`) may be omitted (verified still-green omitted). Verified: build 0/0; `Archive` filter **39/39** (was 26/13); `Category=Fast` **811 passed / 0 failed** (was 798/13 after re8). Lesson: when adding a required field to a `Shared` type, grep **every** mock/fixture JSON (`ArchiveTests.fs` builders + `src/Tests/fixtures/worktrees.json`) and update it — a "timeout" in a mock-API E2E test is almost always a decode failure, not the environment.
- **In-place-morph E2E test made to actually assert (`tm-canvas48-hmb`):** `CanvasPaneTests."Canvas iframe updates when contentHash changes"` was feature-introduced but **always** hit `Assert.Inconclusive` (it modified an on-disk fixture canvas file that never exists — fixture worktree paths are synthetic, e.g. `Q:/code/TestProject/feature-multidoc`, so `CanvasDocServer` returns 404 and no file is ever there to edit) and, worse, its real assertion expected the iframe `src` to **change** on content update — the opposite of the shipped behavior (the doc morphs **in place**: stable `src`, no `?v=` cache-buster, via a `{action:'content-updated'}` postMessage; see Morph scope §3 and `docs/spec/canvas-pane.md`). Rewrote as `"Canvas content update morphs the iframe in place without swapping its src"` (`feature-multidoc`, 3 AgentDocs): it drives the morph **purely from the UI** — re-selecting an already-visited AgentDoc tab dispatches `MorphActiveDoc` (`App.fs SelectCanvasDoc`, `wasAlreadyVisited`) which posts `content-updated` to the active iframe — needing **no** on-disk file and **no** server hash change, both impossible in `--test-fixtures` mode (synthetic paths + static `ContentHash`, so the data-poll morph trigger can never fire). **Judgment call:** the re-select must be a **same-tab** re-click of the *already-active* details tab, not an overview→details switch — `touchVisitedDoc` (`App.fs`) moves the re-selected doc to the **front** of the visited list, so switching the active tab reorders the keyed React iframe children, moving/reloading the iframe DOM node and wiping the Playwright-injected listener (and racing the morph's `querySelector('.canvas-iframe-active')` against React's commit). Re-clicking the active tab keeps it active → no reorder, no reload, no race → the injected listener survives and reliably receives the morph postMessage. Asserts: a `content-updated` message arrives in the details frame (morph fired), the `src` is **identical** before/after (no swap) and carries no `?` cache-buster, and the same frame document persists. Verified: build 0/0; `Category=Canvas` **48/48 pass, 0 skipped** (was 47 passing + this 1 always-inconclusive). Lesson: an E2E test that can only `Assert.Inconclusive` in the very environment it runs in is dead coverage — drive observable product signals (UI-triggered postMessage) instead of unreachable on-disk state, and beware that re-selecting a visited canvas tab reorders the iframe list.
## Resolved Questions

All planning-time open questions are resolved and shipped:

1. **Discriminator → filename-based classifier (`tm-canvas48-cc4`).** `CanvasScanner.scan` sees generic `.html` files and cannot know provisioning origin, so a single shared classifier (`beads.html` → `SystemView`, everything else `AgentDoc`) sets the typed `Kind` field, which is then serialized to the client. Kept generic for future views.
2. **UI → differently-styled entry, not a header button (`tm-canvas48-86v`).** The `SystemView` renders as a distinct entry pinned to the far left of the tab strip, marked with a "BD" glyph + issue-count badge, with no liveness dot and a distinct CSS class (`.canvas-system-tab`, not a normal agent-doc tab).
3. **Other system views → generic from day one (`tm-canvas48-cc4`).** The classifier returns `CanvasDocKind` and is the single place to register future system views (e.g. a CI/build view).
4. **Archive → no (`tm-canvas48-86v`).** The archive button is hidden for `SystemView` (it is server-regenerated, not user-owned). A `SystemView` entry only appears when `beads.html` exists, which provisioning already guarantees only when the beads DB has ≥1 item.
