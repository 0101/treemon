# Canvas System Views — Separating Beads from Agent Docs

> Status: **Proposal / next-feature plan.** Captures the architectural recommendation that came out of debugging the beadspace 30s state-reset (see `docs/spec/canvas-pane.md`, `docs/spec/beadspace-canvas.md`).

## Problem

The beads dashboard (`beads.html`) is **auto-generated by the server** and renders in the canvas pane. Today it flows through the pane as a plain `CanvasDoc` with **zero special-casing in `CanvasPane.fs`**. As a result it inherits the entire generic "canvas document" machinery — including the parts that were designed for a fundamentally different thing.

Two distinct concepts are currently collapsed into one (`CanvasDoc`):

| | **Agent doc** (e.g. a status page, a form) | **Beads dashboard** (a generated view) |
|---|---|---|
| Created by | a coding session writing `.html` | the server, from `BeadspaceTemplate` |
| Owner | the author session (`OwnerSessionId`) | none |
| Where content lives | in the file | in `.beads/beads.db`, fetched via `/beads-data` |
| How it updates | file overwrite → hash change → **morph** | its own 30s data poll + in-place refresh |
| Interactivity | posts payloads back to its owner session | none (only link-nav + self `refresh-beads`) |

### What the beads dashboard actually uses vs. inherits

Layer-by-layer (full diagram was shared in the canvas pane during the session):

- **Genuinely shared (keep):** scan + hash (`CanvasScanner`), serve + inject (`CanvasDocServer` @ :5002), the **pane shell** (tabs, iframe, docking, archive, overview), disk-as-source-of-truth.
- **Beads-specific (already special, fine):** auto-provisioning (`BeadspaceProvisioner`) and the private `/beads-data` JSON endpoint.
- **Inherited but a misfit (the problem):**
  - **Liveness dot** — always shows "dead" because there is no owner session.
  - **`▶ Start session` button** — appears for a generated view, where starting a session is meaningless.
  - **Message bridge** (postMessage → session `/inject`, queue) — unused; there is no author session to route to.
  - **Awareness** (unviewed badge / card notification / idle auto-display keyed on `contentHash`) — largely inert, because the file hash is stable while the *data* changes. Beads "newness" already lives on the worktree card as `BeadsSummary`.
  - **Morph (Level 3)** — redundant with the dashboard's own refresh, and actively harmful when it does fire (a deploy/template change morphs the live, JS-rendered dashboard back down to the empty template shell).

The session-document layer exists for "an interactive document authored and owned by a live session." A beads dashboard is **none of those** — it is a system-generated, data-driven view that only wants the UI real-estate.

## Goals

- Model the two concepts distinctly so misfit behaviors become **unrepresentable** rather than emergent from `OwnerSessionId = None`.
- Give the beads dashboard a **distinct UI affordance** (it should not look like, or sit among, agent doc tabs).
- Preserve the full agent-doc experience (ownership, liveness, Start-session, message bridge, morph, awareness) for genuinely agent-authored docs.
- No regressions to the beadspace fixes already shipped (template sync + unchanged-payload skip-guard).

## Non-Goals

- Removing morph / Level 2–4 — they are valuable for agent docs (interactive, file-driven, overwrite-in-place). They simply should not apply to system views.
- Re-theming or restructuring the beadspace dashboard internals.

## Proposed Approach

### 1. Model the kind (domain-driven, repo-idiomatic)

Add a kind to the canvas doc domain type, e.g.:

```fsharp
type CanvasDocKind =
    | AgentDoc      // authored & owned by a session; interactive; file-driven
    | SystemView    // server-generated; data-driven; no owner (e.g. beadspace)
```

- `CanvasScanner` / provisioning tags `beads.html` (and any future generated view) as `SystemView`; everything else is `AgentDoc`.
  - Simplest discriminator initially: provisioned filename (`beads.html`), or a marker in the served HTML (`<meta name="canvas-system-view">`). Prefer a typed field over filename sniffing in the long run.
- Gate the session-document machinery on `AgentDoc`:
  - Liveness dot, `▶ Start session`, message-bridge routing, and **morph signaling** only apply to `AgentDoc`.
  - Awareness for `SystemView` can be dropped or replaced by data-aware signaling (the card's `BeadsSummary` already covers "something changed").

This makes the misfit states impossible to construct, instead of relying on `None` defaults that happen to render as "dead."

### 2. Distinct UI for system views

- Beads should present as a **dashboard affordance**, not a doc tab — e.g. a dedicated button/icon in the pane header (or worktree card) that opens the beads view, visually separated from the agent-doc tab strip.
- A `SystemView` keeps: the iframe, docking/position, and the pane real-estate. It drops: liveness dot, Start-session, and the doc-style tab treatment.
- Open question: should a worktree's beads view live *inside* the same pane tab strip (styled differently) or be a first-class toggle alongside the canvas button? (Decide during planning.)

### 3. Morph scope

- Morph (and the idiomorph injection) applies to `AgentDoc` only. `SystemView` docs are served without morph signaling (or with the controller present but never told to morph).
- This dissolves the earlier "morph vs beadspace" conflict entirely: beads opts out by virtue of its kind.

## Decisions (carried in from the session)

- **Template sync (shipped):** `beads.html` is regenerated whenever its on-disk content differs from `BeadspaceTemplate.html`, so template fixes propagate on deploy. It is a generated view, not user-customized.
- **Unchanged-payload skip (shipped):** the dashboard's 30s poll skips re-render when `/beads-data` is byte-identical to the prior fetch.
- **Beads canvas = All Issues table only (shipped, `tm-canvas48-dsh`):** the beads `SystemView` renders only the "All Issues" table. The second "Dashboard" view (stat cards, status donut, priority/label/type charts, triage + active-issues panels) and the entire top navigation bar were removed. The single issues view is always visible (`.view { display: block }` — no nav/`.active` toggle), and the detail panel, status/search/filter controls, 30s poll, unchanged-payload skip, and postMessage refresh are all retained.
- **`.fs`/`.html` byte-identical reconciliation (`tm-canvas48-dsh`):** `BeadspaceTemplate.fs` (the `[<Literal>]` served at runtime) and `BeadspaceTemplate.html` (loaded from disk by the E2E tests) had silently diverged — the `tm-canvas48-h8s` `addBadge` detail-panel refactor had landed only in `.fs`. Reconciliation rule going forward: make edits to `.html`, converge the detail panel to the `addBadge` helper, then regenerate `.fs` as `module Server.BeadspaceTemplate` + `let html = """` + the exact `.html` bytes + `"""` so the literal's content is byte-identical to the file (CRLF throughout, no BOM, no `"""` inside the content). No automated test enforces this equality yet; `BeadspaceProvisionerTests` only round-trips the literal.
- **Separate, don't gut:** agent-authored interactive docs are a real use case, so the session-document layer stays — beads is carved out of it.
- **Server injection gated by kind (shipped, `tm-canvas48-dhl`):** `CanvasDocServer.handleCanvasRequest` classifies the served filename via `CanvasDocKind.classify` and builds the `<head>` injection via the new `CanvasDocServer.buildInjection`. Both kinds keep `baseStyle` + `linkInterceptor`. `AgentDoc` keeps the full injection (`bridgeScript` heartbeat + `IdiomorphScript.idiomorphJs` + `IdiomorphScript.morphController`); `SystemView` (e.g. `beads.html`) omits all three. Rationale: a system view must never morph (a morph stomps the live, JS-rendered dashboard back to the empty template shell), and — the one judgment call the task left open — `bridgeScript` is also dropped because a system view has no owner session to route bridge messages to. Safe because the dashboard drives itself via its own 30s `/beads-data` poll and a direct `refresh-beads` postMessage listener, neither of which depends on the bridge heartbeat (which only dispatches `canvasBridgeMessage` events the dashboard never consumes). Covered by `CanvasDocServerTests` (`buildInjection`/`classify` mapping); the end-to-end HTTP proof against :5002 is owned by the `tm-canvas48-klm` verify task.
- **Client session-document machinery gated by kind (shipped, `tm-canvas48-fj9`):** the client mirrors the server gating so misfit affordances never render for a `SystemView`. `CanvasPane.fs` routes both liveness-dot call sites (tab strip + overview) through `livenessDotFor`, which returns `Html.none` for `SystemView`, and gates the `▶ Start session` button on `doc.Kind = AgentDoc`. `App.fs` adds a shared `canvasDocKind repos scopedKey filename` lookup and uses it to (a) drop `SystemView` docs from idle auto-display candidates so a beads data-poll never focus-steals, and (b) suppress the `MorphActiveDoc` signal for `SystemView` active docs. Judgment call: the task scoped morph-gating to the data-poll `morphCmd`, but the same `Cmd.ofMsg MorphActiveDoc` is also emitted on tab re-select (`SelectCanvasDoc`, `wasAlreadyVisited`), so **both** emit sites were gated to fully realize "morph applies to AgentDoc only" on the client (the re-select morph was already inert for `SystemView` since dhl injects no morph controller, so this only removes a redundant no-op postMessage). Covered by three E2E render assertions in `CanvasPaneTests` against the multirepo fixture (beads `SystemView` + dashboard/status `AgentDoc`): SystemView shows no liveness dot (tab + overview) and no Start-session, AgentDocs still show both.
- **Awareness excludes SystemView at the data source (shipped, `tm-canvas48-45p`):** content-hash awareness must ignore `SystemView` docs — the beads file hash is stable while its data changes (so it would never signal real newness) and would morph-signal spuriously on a template/deploy edit; beads "newness" already lives on the worktree card as `BeadsSummary`. Implemented as a single private `awarenessDocs = List.filter (fun d -> d.Kind = AgentDoc)` chokepoint in `CanvasAwareness.fs`, routed through `seedLastViewedHashes`, `unviewedDocsByScopedKey`, `canvasHashesByScopedKey`, and `findMostRecentChangedDoc`. `detectCanvasEvents` needs no change of its own: it derives events purely from the hash maps, and since `canvasHashesByScopedKey` never emits a `SystemView` hash, a beads data-change can't produce a `NewDoc`/`UpdatedDoc` event. This also makes `findMostRecentChangedDoc` self-defending — `App.fs` already filters auto-display candidates to `AgentDoc` (fj9), but the function now ignores `SystemView` regardless of caller. Covered by `SystemViewAwarenessTests` in `CanvasAwarenessTests.fs` (11 tests across all four functions, proving a `SystemView` doc never contributes to unviewed counts, canvas events, seeded hashes, or the idle auto-display target). Note: `CanvasAwarenessTests` has 9 pre-existing failures (4 `Auto-display*` + 5 `MarkDocViewed*`) from the `App` `.cctor` building a Fable.Remoting client proxy under the .NET test host — unrelated to awareness, documented in `tm-canvas48-cc4`; the new tests avoid all `App` module bindings and so are immune.
- **Distinct SystemView affordance + archive gating (shipped, `tm-canvas48-86v`):** the `SystemView` (beads) now renders as a distinct entry in the tab strip via a new private `CanvasPane.systemViewTab` — class `.canvas-system-tab` (NOT `.canvas-tab`), a `"BD"` glyph plus a count badge (`.canvas-system-tab-count`), no liveness dot. The label is the worktree's total beads issues `Open + InProgress + Blocked + Closed` (helper `beadsTotal`, sourced from the in-scope `wt.Beads`), replacing the dashboard's former top-bar "N issues" meta. It is pinned far-left by a stable `List.sortBy` keying `SystemView → 0`/`AgentDoc → 1` over `wt.CanvasDocs`. **Gate change:** the tab strip now renders when `wt.CanvasDocs.Length > 1 || hasSystemView` (was `> 1`), so a lone `beads.html` still shows its count badge; a lone `AgentDoc` still shows no tabs. The **archive button** is gated in `headerBar` by widening its parameter from `activeFilename: string option` to `activeDoc: CanvasDoc option` and rendering the button only for `Some d when d.Kind = AgentDoc` (a `SystemView` is server-regenerated, not user-owned). CSS lives in `index.html` (`.canvas-system-tab*`, no inline styles). Judgment call: the fj9 test `SystemView tab shows no liveness dot…` located the beads entry via `.canvas-tab` HasText "beads", which my change makes a no-op (beads is no longer a `.canvas-tab` and its label is the count, not "beads"); its `beadsTab` locator was repointed to `.canvas-system-tab` to keep the assertion honest. Covered by three new `CanvasPaneTests` E2E assertions against the multirepo fixture (beads total = 2): distinct leftmost non-`.canvas-tab` entry with no liveness dot, count badge shows "2", archive hidden while the SystemView is active but present after switching to an `AgentDoc`. Verified: full build 0/0; Fable compile clean; 37/38 Canvas E2E pass (the lone failure, `Liveness dot renders in overview entries`, is the pre-existing `.canvas-overview-branch` selector bug from cc4, untouched here).

## Open Questions (resolved during planning)

1. **Discriminator → filename-based classifier.** `CanvasScanner.scan` sees generic `.html` files and cannot know provisioning origin, so a single shared classifier (`beads.html` → `SystemView`, everything else `AgentDoc`) sets the typed `Kind` field, which is then serialized to the client. Kept generic for future views.
2. **UI → differently-styled entry, not a header button.** The `SystemView` renders as a distinct entry pinned to the far left of the tab strip, marked with a "BD"/task glyph, with no liveness dot and a distinct CSS class (not a normal agent-doc tab).
3. **Other system views → generic from day one.** The classifier returns `CanvasDocKind` and is the single place to register future system views (e.g. a CI/build view).
4. **Archive → no.** The archive button is hidden for `SystemView` (it is server-regenerated, not user-owned). A `SystemView` entry only appears when `beads.html` exists, which provisioning already guarantees only when the beads DB has ≥1 item.
